# gov-cli

Sleeping container that holds `cardano-cli` plus the Antithesis composer
**governance driver scripts**. Antithesis `docker exec`s the drivers
per tick to perform governance operations against `relay1` in the
`cardano_node_governance` testnet.

Each *logical* cardano-cli governance operation is its own composer
driver:

| Driver | Prefix | Operation |
|---|---|---|
| `first_setup` | `first_` | one-shot setup: authorize CC hot keys, register DReps, delegate vote-stake (no faults) |
| `parallel_driver_create_action` | `parallel_driver_` | submit one InfoAction |
| `parallel_driver_vote` | `parallel_driver_` | cast DRep + SPO + CC votes on a pending action |
| `anytime_govstate_invariant` | `anytime_` | gov-state is always well-formed |
| `eventually_votes_recorded` | `eventually_` | post-fault: an action carries votes |
| `finally_governance_summary` | `finally_` | end-of-run coverage marker |

## Two driver languages

Both sets implement the same drivers and emit the same `sdk.jsonl`:

- `composer/governance-bash/` — `cardano-cli` wrapped in shell.
- `composer/governance-python/` — the standalone
  [`cardano-clusterlib`](https://github.com/IntersectMBO/cardano-clusterlib-py)
  library (which wraps `cardano-cli`) driving the same flow. No
  cardano-node-tests harness is used — only the wrapper library's core
  functions.

Select one at build time; it becomes the live composer template at
`/opt/antithesis/test/v1/governance`:

```sh
docker build --build-arg DRIVER_LANG=bash   -t gov-cli:bash   .
docker build --build-arg DRIVER_LANG=python -t gov-cli:python .
```

Both sets are also copied to `/opt/gov-drivers/` in the image for
inspection.

## Assets and mounts

The drivers consume assets generated by `gov-configurator` (cardonnay's
`conway_fast` genesis + `governance_data`):

| Mount | Purpose |
|---|---|
| `relay1-state:/state:ro` | `node.socket` to query/submit against relay1 |
| `gov-data:/gov-data` | `governance_data/` (DRep/CC/vote-stake keys + certs), `faucet/`, `pools/node-pool*/cold.*`, plus writable `state/` for driver coordination |
| `gov-state:/gov-state:ro` | shelley genesis so `cardano-clusterlib` can read the network magic (python set) |
| `tmpfs:/work` | scratch for tx building |

## Coordinating create and vote — the chain is the queue

`create` and `vote` are independent, **stateless** `parallel_driver_*`
commands. There is no local coordination ledger: the vote driver reads
the live set of governance actions straight from the node via an N2C
`gov-state` query (`live_info_actions`), RNG-picks one, and votes. An
action leaves the votable set only when the **ledger** expires or enacts
it — so a transient submit/lock failure is **self-healing** (the action
is simply picked again next tick), and there is no "retire" to corrupt.

`create` just submits an InfoAction; it appears in `gov-state` on
confirmation. The two drivers never talk to each other — the system
under test holds all the state.

> Earlier this used two append-only logs (`created.log` / `rejected.log`)
> and a `created − rejected` difference. A live Antithesis run showed
> that under a block-production stall, transient faucet-lock/submit
> failures were misclassified as rejections and **permanently retired
> still-valid actions**. Selecting from `gov-state` (relay1 is
> fault-excluded, and a LocalStateQuery answers even during a stall)
> removes that failure class by construction.

Every choice the vote driver makes is handed to the **Antithesis RNG**
(`antithesis_random`, falling back to `/dev/urandom` locally): which
live action, which voter (a random DRep, SPO or CC member), and **yes,
no or abstain**. So each invocation is one voter casting one random vote
on one random live action, and the hypervisor explores the (action,
voter, decision) space.

The only writable state left under `gov-data/state/` is the faucet spend
lock (all txs draw from the single genesis-utxo), the perturbation
probe's `chain_verdict`, and the setup marker.
